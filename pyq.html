👥 Unit 1: Organizational Behavior, Staffing & Team Dynamics
Q1
Explain the Organizational Behavior approach to software staffing and team dynamics.
📌 Definition

Organizational Behavior (OB) is the study of how individuals and groups act within an organization. In software projects, OB principles help managers select, motivate, and organize teams effectively to maximize productivity and project success.

🎯 Staff Selection

Hire based on technical skills + communication ability + problem-solving + team compatibility. Interview for attitude, not just coding ability.

🏆 Motivation Theories

Maslow's Hierarchy: salary → security → social → recognition → growth. Manager must address lower needs before higher ones motivate.

🧠 McGregor Theory X/Y

Theory X: employees lazy, need supervision. Theory Y: employees self-motivated, want responsibility. Software teams = Theory Y approach.

👥 Team Dynamics

Tuckman stages: Forming → Storming → Norming → Performing → Adjourning. Manager must guide team through each stage!

📊 Tuckman's Team Development Model
👋 Forming — Team meets, polite, cautious. Roles unclear. PM must define goals clearly.
⬇
⚡ Storming — Conflict over roles, methods, ideas. Most challenging phase! PM = mediator.
⬇
🤝 Norming — Team agrees on norms, processes. Cohesion builds. Conflict reduces.
⬇
🚀 Performing — High productivity! Team works autonomously. PM delegates freely.
⬇
🏁 Adjourning — Project ends, team disbands. Celebrate success, conduct lessons-learned review.
✏️ Exam Tip: Always draw Maslow's pyramid AND Tuckman's 5 stages. Mention Theory X vs Y. Conclude: "OB principles help PM build high-performing teams aligned with project goals."
Q2
What is the Oldham–Hackman model? How does it help in enhancing team performance?
📌 Definition

The Hackman & Oldham Job Characteristics Model (1976) states that job design itself motivates people. If a job has 5 key characteristics, employees will be intrinsically motivated without external rewards. Used to redesign software development jobs for maximum motivation.

🎯
1. Skill VarietyJob uses many different skills. Not repetitive boring work.
📦
2. Task IdentityComplete a visible, whole piece of work from start to finish.
🌍
3. Task SignificanceWork impacts others meaningfully. Matters to users/society.
🎮
4. AutonomyFreedom to decide HOW to do the job. Independence in work.
💬
5. FeedbackClear, direct information on performance quality.

📐 MPS Formula — Motivating Potential Score

MPS = [(Skill Variety + Task Identity + Task Significance) / 3] × Autonomy × Feedback
Higher MPS = More motivating job. If ANY of Autonomy or Feedback = 0 → MPS = 0!
Example for System Testing Group: Give tester ownership of entire test module (↑Task Identity), let them choose testing tools (↑Autonomy), share defect reports with customer (↑Feedback, ↑Task Significance)

⭐ Applying to Low Job Satisfaction Departments

System Testing Group: Let testers own entire features end-to-end (↑Task Identity). Share quality reports with stakeholders (↑Significance + Feedback).
Help Desk: Rotate to varied problems (↑Skill Variety). Allow staff to resolve issues independently (↑Autonomy). Track + share resolution metrics (↑Feedback).
Batch Input: Assign specific data domains (↑Task Identity). Automate routine parts, focus on complex validation (↑Skill Variety).

🧩 Memory: 5 characteristics = STTAF → "Smart Teams Take Appropriate Feedback" | MPS = first 3 averaged × last 2 multiplied

Q4
Three mental obstacles to good decision-making: Faulty heuristics, Escalation of commitment, Information overload. Steps to reduce danger of each.

❌ 1. Faulty Heuristics (Mental Shortcuts)

Using rules of thumb that are systematically biased. E.g., anchoring bias (first number heard influences all estimates), availability bias (recent events seem more probable).
Steps to Reduce: Use structured estimation methods (PERT, COCOMO), Delphi method (independent estimates), require evidence-based decisions, devil's advocate role in reviews.

⬆️ 2. Escalation of Commitment

Continuing to invest in a failing project because of sunk costs. "We've already spent so much — we can't stop now!" Common in late, over-budget projects.
Steps to Reduce: Set predefined "kill criteria" at project start, use independent reviews at milestones, separate decision-makers from original project champions, calculate EAC honestly.

📊 3. Information Overload

Too much data = poor decisions. Managers drown in reports, metrics, emails, and miss the critical signals. Analysis paralysis.
Steps to Reduce: Dashboard with key metrics only (KPIs), exception reporting (only alert when threshold breached), scheduled decision meetings (not ad-hoc), information hierarchy (what each role needs to see).

✏️ Exam Tip: For each obstacle: define it clearly → give a software project example → state 3 steps to reduce it. This structure = full marks.

Sprints (1–4 weeks). Daily standups. Product backlog → Sprint planning → Sprint review → Retrospective. Roles: Product Owner, Scrum Master, Dev Team.

Q2
What is the COCOMO II model? Explain its components and how it is used in software cost estimation.
📌 COCOMO II — Definition

COCOMO II (Constructive Cost Model II) by Barry Boehm (1995) is an improved parametric model for estimating software development effort, schedule, and cost. Unlike COCOMO I (which uses only LOC), COCOMO II uses complexity, scale factors, and effort multipliers making it more accurate for modern software development.

📐 COCOMO II Formula

Effort (PM) = A × (Size)E × ∏EMi
Where A = 2.94, E = B + 0.01×ΣSF (B=0.91), Size in KSLOC or Function Points

Duration (Months) = C × (Effort)F Where C = 3.67, F = D + 0.2×(E-B) (D=0.28)

EMi = Effort Multipliers (17 cost drivers rated Very Low/Low/Nominal/High/Very High)

🏛️ COCOMO Hierarchy — 3 Levels
LevelInputsAccuracyUse When
🔵 BasicKLOC only, 3 project modes±50–100%Feasibility / rough estimate
🟠 IntermediateKLOC + 15 cost drivers±25–50%After requirements phase
🔴 Detailed/CompleteCost drivers applied per module+phase±10–25%After design — full baseline

📊 5 Scale Factors

Precedentedness, Development flexibility, Architecture risk resolution, Team cohesion, Process maturity. Each rated 0–5. Sum × 0.01 added to exponent.

🔧 17 Cost Drivers

4 categories: Product (RELY, DATA, CPLX), Platform (TIME, STOR), Personnel (ACAP, PCAP, AEXP), Project (TOOL, SITE, SCED). Each is a multiplier 0.70–1.65.

📐 3 Sub-Models

Application Composition (early prototyping), Early Design (after architecture), Post-Architecture (most detailed, accurate). Each suits different project stage.

COCOMO I

  • Uses only KLOC (size)
  • Three modes: Organic/Semi/Embedded
  • Basic formula: E=a×(KLOC)^b
  • Less accurate, simpler
  • 1981 — waterfall era

COCOMO II

  • Uses Size + Scale Factors + 17 Multipliers
  • Function Points supported
  • Multiple sub-models
  • More accurate, complex
  • 1995 — modern development
Q3
What are Full Function Points in the COSMIC model? How do they contribute to estimation? Outline COSMIC FFP for software cost estimation.
📌 COSMIC FFP — Definition

COSMIC (Common Software Measurement International Consortium) Full Function Points is an ISO standard (ISO 19761) for measuring functional software size. It counts data movements through software rather than lines of code — making it language-independent and objective. Each data movement = 1 CFP (COSMIC Function Point).

➡️ ENTRY (E)

Data moves FROM user/external INTO software. E.g., user submits form, file uploaded, sensor reading. User → System.

⬅️ EXIT (X)

Data moves FROM software TO user/external. E.g., report displayed, email sent, error message shown. System → User.

📖 READ (R)

Data moves FROM persistent storage TO software. E.g., retrieve customer from DB, load config from file. Storage → System.

✏️ WRITE (W)

Data moves FROM software TO persistent storage. E.g., save order to DB, update record, delete. System → Storage.

⭐ COSMIC Counting Example

Feature: User Login
1. User enters username+password → ENTRY = 1 CFP
2. System reads user record from DB → READ = 1 CFP
3. System reads session/permissions → READ = 1 CFP
4. System writes session record to DB → WRITE = 1 CFP
5. System shows welcome screen → EXIT = 1 CFP
Total = 5 CFPs for Login feature

Contribution to Estimation: Total CFPs → ÷ industry productivity rate (CFPs/person-month) → Effort estimate. More CFPs = larger system = more effort.

🧩 Memory: COSMIC 4 types = EXRW = "Everyone Exits Reading Wildly" | Entry=User→System, Exit=System→User, Read=Storage→System, Write=System→Storage

What is Critical Path Method (CPM)? How is it used to identify project bottlenecks? Discuss Forward Pass and Backward Pass.
📌 CPM — Definition

CPM is a network-based project scheduling technique that identifies the longest path through a project network. This "critical path" determines the minimum project duration. Any delay on this path = delay to entire project. Activities NOT on critical path have "float" = spare time.

➡️ Forward Pass — Earliest Times

  • Go LEFT to RIGHT through network
  • EF = ES + Duration
  • Multiple predecessors → ES = MAX of all EFs
  • Finds: Earliest Start (ES) and Earliest Finish (EF)
  • Result: Earliest possible project completion date

⬅️ Backward Pass — Latest Times

  • Go RIGHT to LEFT through network
  • LS = LF − Duration
  • Multiple successors → LF = MIN of all LSs
  • Finds: Latest Start (LS) and Latest Finish (LF)
  • Result: Latest each task can start without delaying project

📐 Key CPM Formulas

Total Float = LS − ES = LF − EF
Free Float = ES(next activity) − EF(current activity)
Critical Path = all activities where Total Float = 0
Project Duration = EF of last activity on critical path
Crashing Cost Slope = (Crash Cost − Normal Cost) / (Normal Duration − Crash Duration)

✏️ Exam Tip: Always draw a table with columns: Activity | Duration | ES | EF | LS | LF | Float | Critical? Then state: "Critical path = [path where Float=0]". Draw the node boxes with 6 values each.
Q3
Write down about: (i) Risk Identification (ii) PERT Technique.
⚠️ (i) Risk Identification

Risk identification is the first step of risk management — systematically finding everything that could go wrong in a software project before it happens.

🧠 Brainstorming

Team generates risks freely. No criticism during session. PM facilitates. Capture everything on whiteboard first.

✅ Checklists

Standard risk lists from past projects/industry experience. Quick scan of known risk categories.

👤 Expert Interviews

Interview experienced stakeholders. Learn from past project mistakes. Domain experts identify technical risks.

🔍 SWOT Analysis

Strengths/Weaknesses/Opportunities/Threats. Strategic view of project risks from multiple angles.

⭐ Risk Categories in Software Projects

👥 People risks — key staff leaving, illness, skill gaps
📋 Requirements risks — unclear/changing requirements, scope creep
💻 Technology risks — new technology, integration failures
💰 Estimation risks — optimistic schedules, underestimated complexity
🏢 Organisational risks — budget cuts, priority changes, restructuring

📊 (ii) PERT Technique — Program Evaluation and Review Technique

PERT is a probabilistic scheduling technique using 3 time estimates per activity. Better than CPM for projects with uncertain durations (R&D, software).

📐 PERT Formulas

Expected Time: TE = (O + 4M + P) / 6
Standard Deviation: σ = (P − O) / 6
Variance: σ² = [(P − O) / 6]²
Project Variance = Sum of variances of critical path activities
Z = (Target Date − TE) / σ → use Z-table for probability

O = Optimistic (best case) | M = Most Likely | P = Pessimistic (worst case)

⭐ PERT vs CPM

CPM = single time estimate (deterministic). PERT = 3 estimates (probabilistic). CPM = construction projects. PERT = R&D/software (uncertain). PERT gives completion probability. CPM focuses on cost-time crashing trade-off.

💹 Numerical 2: Earned Value Management Problems
EVM1
Calculate EAC and VAC when both SPI and CPI influence the project.
BAC=$22,000 | EV=$13,000 | PV=$14,000 | AC=$15,000

📌 Step 1: Calculate Basic EVM Metrics

SV (Schedule Variance) = EV − PV = 13,000 − 14,000 = −$1,000 (Behind Schedule ⚠️)
CV (Cost Variance) = EV − AC = 13,000 − 15,000 = −$2,000 (Over Budget ⚠️)
SPI (Schedule Performance Index) = EV/PV = 13,000/14,000 = 0.929 (<1 = behind schedule)
CPI (Cost Performance Index) = EV/AC = 13,000/15,000 = 0.867 (<1 = over budget)

📌 Step 2: EAC — When BOTH SPI and CPI influence

EAC = BAC / (CPI × SPI) = 22,000 / (0.867 × 0.929) = 22,000 / 0.806 = $27,295
This means: project is now forecast to cost $27,295 instead of the budgeted $22,000
EAC = $27,295 (Revised total cost estimate)

📌 Step 3: VAC — Variance At Completion

VAC = BAC − EAC = 22,000 − 27,295 = −$5,295
Negative VAC = project will be over budget at completion by $5,295
VAC = −$5,295 ⚠️ Project will EXCEED budget by $5,295 at completion!
